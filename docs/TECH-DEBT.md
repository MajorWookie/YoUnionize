# Known Tech Debt

## High Priority

- [ ] **No rate limiting on API endpoints** — RAG queries and compensation analysis are computationally expensive with no throttling.
- [ ] **Email confirmations disabled** in Supabase auth config — anyone can sign up with any email.
- [ ] **Display layer reads filing_summaries.ai_summary for per-item content** — After the 2026-04-29 per-section rewrite, per-item summaries (Risk Factors, MD&A, etc.) live on `filing_sections.ai_summary` instead. `CompanySummaryCard` and the `company-detail` Edge Function need updates to merge per-section rows into the response payload.

## Medium Priority

- [ ] **No structured logging** — All logging via `console.info()` with no request IDs, user context, or JSON structure. Hard to search in production.
- [ ] **No E2E tests in CI** — Playwright scaffolded but no test files or CI integration.
- [ ] **No SAST/dependency scanning** in CI pipeline.
- [ ] **Section-level review CLI + Edge Functions deferred** — Per-section review state landed in the schema (2026-04-29) but the `review section-list/verify/edit` commands and matching `review-section-*` Edge Functions are pending a follow-up branch. Filing-level review still works.
- [ ] **Specialized prompts deferred** — Dispatch table reserves slots for `cybersecurity`, `controls_and_procedures`, `related_transactions` prompt kinds, but they currently route through the generic `narrative` prompt label. Specialized prompt files in `packages/ai/src/prompts/` are pending a follow-up branch.
- [ ] **DEF 14A narrative item extraction is unreliable in sec-api.io** — sec-api's `/extractor` endpoint is built for the rigid 10-K/10-Q "Part X Item Y" item taxonomy, but DEF 14A proxy statements are free-form ("Compensation Discussion & Analysis", "Director Nominees", "Pay vs Performance", etc.). For many filings, asking for `part1item1` or `part1item7` returns HTTP 404. Confirmed 2026-04-29 with ORCL DEF 14A `0001193125-25-220801` — both items 404'd despite a valid `linkToFilingDetails` URL. The client now treats 404 as `fetch_status='empty'` rather than `'error'` (a 404 means "this item isn't in this filing", semantically empty), so DEF 14A rows no longer pollute the error queue. The narrative content is still missing, though — to actually populate proxy summaries we'd need to: (a) drop DEF 14A from the section dispatch and rely solely on the structured `/compensation` endpoint we already use; (b) fall back to full-text scraping of `linkToFilingDetails` with a heading-based parse; (c) accept that DEF 14A summaries are exec-comp-only and skip the narrative. Affects ~2 rows per DEF 14A across the corpus.
- [ ] **Rollup PROMPT_VERSIONS bumped to v3 — backfill is opt-in** — PR 5b (2026-04-29) bumped `CURRENT_SUMMARY_VERSION` 2 → 3 alongside the rollup prompt rewrites. The pipeline's "needs work" filter is `summary_version IN (0, -1)`, so existing v2 rows stay put — they don't auto-re-run under the new prompts. To backfill a company: `UPDATE filing_summaries SET summary_version = 0 WHERE company_id = '<uuid>' AND filing_type IN ('10-K', '10-Q');` then run `bun run scripts/summarize-all.ts --tickers=<TICKER>`. At 50 RPM Tier-1 Claude and ~3 calls per 10-K, full corpus backfill is hours-to-days. Recommended: backfill per-company on demand and monitor Anthropic spend during the rollout window.
- [ ] **`mergeEmployeeImpact` writes a literal "Not analysed for this filing." sentinel when one prompt fails** — `src/server/services/summarization-pipeline.ts` (PR 5b, 2026-04-29) merges `EmployeeOutlookResult` and `WorkforceSignalsResult` into the v2-shape `EmployeeImpactResult` for frontend compatibility. If either source prompt throws, its fields are filled with the literal string "Not analysed for this filing." so consumers don't see `undefined`. The frontend currently renders that string verbatim. Consider either: (a) detect the sentinel in the UI and render a "(this signal wasn't analysed)" affordance; (b) change the merge to omit failed-source fields entirely and update consumers to handle missing keys.
- [ ] **Workforce-signals prompt unverified against real filings** — `packages/ai/src/prompts/workforce-signals.ts` (PR 5b, 2026-04-29) lifted the workforce-geography + H-1B/visa-dependency portion of the old employee-impact prompt into a dedicated prompt that runs on raw section text. The output schema is the same as before, but no real-corpus validation has been done since the split. `scripts/validate-rollup-rewrite.ts` provides a side-by-side judge harness; needs to be run with real credentials before declaring PR 5b's changes battle-tested.

## Low Priority

- [ ] **SST config references One Framework** — `sst.config.ts` still sets `ONE_SERVER_URL` env var. Should be removed or renamed post-migration.
- [ ] **Tamagui v2.0.0-rc.15 is pre-release** — API may change. Monitor for stable release (works currently).
- [ ] **API client error parsing** — `fetchWithRetry()` assumes JSON error responses; HTML error pages (502) will cause secondary parse failures.
- [ ] **Summarization text chunking** — Hardcoded 4 chars/token ratio is approximate; overlap of 50 tokens may lose context at chunk boundaries.
- [ ] **Lambda ingestion handler still wired in `sst.config.ts`** — `src/server/lambda/ingestion.ts` exposes `fetchHandler`/`processHandler` as Lambda entry points, paralleling the live Edge Function 2-phase pipeline. Decide whether to keep an AWS deployment path or retire the lambda + sst.config entries.
- [ ] **`SECTION_LABELS.executive_compensation` may be redundant after embedding dedup** — `src/server/services/summarization-pipeline.ts` defines a friendly label that becomes redundant whenever PR 3's content-hash dedup (2026-04-29) skips the rollup `analysis` chunk in favour of its source section's chunk. Review whether the label is reachable; if not, drop it.
- [ ] **Rollup helpers exported only for the validation harness** — `buildAggregatedContext`, `loadAllSectionSummariesByCode`, `findSectionTextByPromptKind`, `mergeEmployeeImpact` in `src/server/services/summarization-pipeline.ts` are now `export`-ed solely so `scripts/validate-rollup-rewrite.ts` can use the production logic. Once the validation harness is retired (i.e., PR 5b is fully in production), consider de-exporting these or moving them into a separate `rollup-context.ts` module with its own unit tests.

- [x] **~~Embedding chunks duplicated across section + rollup writes~~** — Resolved (2026-04-29, PR #27). `generateAllEmbeddings` now tracks section-chunk content hashes and skips rollup chunks whose hash matches. The DB-level dedup at the insert site is kept as second-line defence. Most concrete win: after PR #26, the exec_comp rollup `.analysis` field is byte-equivalent to the part1item7 section summary, so the rollup chunk now produces zero embedding work.
- [x] **~~`/ask` retrieved per-section embeddings but resolved them in the wrong table~~** — Resolved (2026-04-29, PR #28). `supabase/functions/ask/index.ts` looked up `metadata.section` against `filing_summaries.aiSummary` (rollup keys only), but per-section embeddings carried `metadata.section` set to prompt kinds like `risk_factors` / `mda`. `extractSectionText(undefined)` returned `''` and the candidate was silently dropped — RAG quality rested almost entirely on rollup embeddings. New `resolveEmbeddingText` helper branches on `metadata.sectionCode`: per-section embeddings hit `filing_sections.ai_summary` via the `(filing_id, section_code)` unique index; rollups keep the existing path. Per-section embedding writes also include `metadata.filingSectionId` going forward (existing rows continue to work via composite-key fallback).
- [x] **~~Embedding metadata could carry `filingSectionId` UUID~~** — Resolved (2026-04-29, PR #28). Per-section embedding inserts now include `metadata.filingSectionId = filing_sections.id`. Existing rows continue to work via the `(filingId, sectionCode)` fallback inside `resolveEmbeddingText`.
- [x] **~~`sectionMapByCode` built from pending sections only — broke resumed-run rollups~~** — Resolved (2026-04-29, PR #29). The map was built from `sectionRows` (rows still marked pending, `summary_version IN (0, -1)`), so on a resumed run where some sections finished in a prior invocation those rows were filtered out and the rollup-input map silently missed `risk_factors` / `mda` text from earlier runs. Replaced the in-memory filter with a `loadAllSectionTextByCode` DB query that returns every successfully-extracted section. Fresh runs see the same data; resumed runs now see all sections.
- [x] **~~Rollups re-fed full SEC `rawData` to Claude despite per-section summaries existing~~** — Resolved (2026-04-29, PR #30). `generateCompanySummary` and `generateEmployeeImpact` were passing `filing.rawData` (multi-MB SEC blob) instead of consuming the cached `filing_sections.ai_summary` rows produced earlier in the same run. New `buildAggregatedContext` helper composes a markdown view of the per-section summaries + structured XBRL tables, fed to both rollups in place of `rawData`. Token cost per rollup drops ~80–90%. Validation harness at `scripts/validate-rollup-rewrite.ts` ships alongside.
- [x] **~~`employee_impact` was a kitchen-sink prompt~~** — Resolved (2026-04-29, PR #30). The geography + H-1B/visa portion was lifted into a focused `workforce-signals` prompt that runs on raw `business_overview` + `risk_factors` text (direct quotes/numbers matter for those signals). Outputs from the two prompts are merged into a single `EmployeeImpactResult` shape on `filing_summaries.ai_summary.employee_impact` so frontend consumers stay unchanged.
